# NPM Dependencies Strategy: Eliminating Technical Debt

## Проблема: Зачем мы подавляем warnings?

### Текущее состояние
В `.npmrc` мы используем:
```
legacy-peer-deps=true
```

Это **подавляет симптомы**, но **не решает проблему**.

---

## Анализ: Что скрывается за warnings?

### 1. **Deprecated пакеты** (устаревшие)
В развертывании мы видели более 20+ предупреждений вида:
- `npm WARN deprecated rimraf@3.0.2`
- `npm WARN deprecated inflight@1.0.6` 
- `npm WARN deprecated glob@7.2.3`
- И многие другие...

**Что это означает:**
- Пакеты больше не поддерживаются автором
- Уязвимости безопасности могут не исправляться
- Новые версии Node.js могут быть несовместимы

### 2. **Peer Dependencies конфликты**
Вызвано использованием `legacy-peer-deps=true`:
- NestJS v10 требует определенные версии зависимостей
- Некоторые пакеты в `node_modules` несовместимы
- npm молча устанавливает несовместимые версии
- **Это может привести к runtime ошибкам в production**

---

## Почему это опасно?

### Сценарий 1: Security vulnerability
```
❌ старый пакет с уязвимостью
  └─ npm скрывает warning
  └─ код работает в development
  └─ production падает при использовании функции
```

### Сценарий 2: Node.js версия обновилась
```
❌ deprecated пакет несовместим с Node 22
  └─ все warnings подавлены
  └─ локально работает (старая Node)
  └─ production падает (новая Node 22)
```

### Сценарий 3: Скрытая несовместимость
```
❌ Пакет A требует lodash@4.x
❌ Пакет B требует lodash@3.x
└─ npm устанавливает обе версии (плохо)
└─ код случайно использует неправильную версию
└─ behavior отличается между средами
```

## Действующие Deprecated пакеты в ARQ

На основе лога развертывания (есть 20+ warnings):

### High Priority (обновить срочно)
```json
"rimraf": "^5.0.5"          // используется для очистки dist
"@types/node": "^20.10.0"   // может быть несовместима с Node 22
```

### Medium Priority  
```json
"passport-local": "^1.0.0"  // старая версия
"class-validator": "^0.14.0" // есть новая версия 0.14.1+
```

### Low Priority
```json
"swagger-ui-express": "^5.0.0" // может иметь уязвимости
```

## Решение: Правильный подход

### Опция 1: Обновить все зависимости (Рекомендуется)

**Шаги:**

1. **Удалить подавление warnings**
   ```diff
   # .npmrc
   - legacy-peer-deps=true
   ```

2. **Обновить зависимости**
   ```bash
   npm update
   npm install
   ```

3. **Исправить peer dependency конфликты вручную**
   - Проверить требования каждого пакета
   - Выбрать совместимые версии
   - Обновить package.json

4. **Запустить тесты**
   ```bash
   npm test
   npm run build
   ```

### Опция 2: Selective approach (если спешим)

**Для каждого deprecated пакета:**

1. Проверить есть ли обновление
2. Если нет - заменить на активный fork
3. Если критично - оставить с документацией

Пример:
```json
// ❌ Старый, не поддерживается
"passport-local": "^1.0.0"

// ✅ Новый, активно поддерживается  
"passport-local": "^1.0.0" // актуально поддерживается
```

## Примерный план обновления

### Phase 1: Идентифицировать конфликты (1-2 часа)
```bash
npm install --save-exact  # заморозить версии
npm ls | grep deprecated  # найти все deprecated
npm audit                  # найти уязвимости
```

### Phase 2: Обновить критичные (3-4 часа)
```bash
# Обновить Node.js type definitions
npm install --save-dev @types/node@latest

# Обновить build tools
npm install --save-dev rimraf@latest typescript@latest
```

### Phase 3: Тестировать (2-3 часа)
```bash
npm run build
npm test
npm run build:prod
```

### Phase 4: Развернуть (1 час)
```bash
git commit -m "fix: Update npm dependencies and remove legacy-peer-deps"
git push  # CI/CD обновит
```

## Вывод

**Мы подавляем warnings потому что:**
1. Проблема была срочной (npm ci не работал)
2. Нужна была быстрая доставка (deployment working)
3. Настоящее исправление требует времени

**Это нужно исправить потому что:**
1. Production безопасность
2. Будущая совместимость
3. Long-term поддерживаемость

**Рекомендация:**
- В следующую итерацию - посвятить время на обновление
- Это улучшит стабильность всего проекта
- ROI высокий (1-2 дня работы, месяцы выигранной стабильности)
